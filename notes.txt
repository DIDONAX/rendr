general notes for the future:

OpenGl:
    - is a state machine, client server architecture -> behaviour depends on the current state set, state changes can be expensive -> keep state mutations to the minimum.
    - example of data that can be assosiated to a vertex: position, color , normal texture coords
    - commands processed are in order
    - There are two classes of framebuffers: a window system-provided framebuffer
    associated with a context when the context is made current, and application-created
    framebuffers. The window system-provided framebuffer is referred to as the default framebuffer. Application-created framebuffers, referred to as framebuffer objects, may be created as desired, A context may be associated with two framebuffers, one for each of reading and drawing operations. The default framebuffer
    and framebuffer objects are distinguished primarily by the interfaces for configuring and managing their state.
Design goals:
  core features:
    - Highthrougtht put batch rendereing
    - minimal driver overhead
    - should offer optimised dynamic data streaming out of the box (example: positions attributes changin each frame)
    - meant to be a building block/kernel for other case specific renderers
    - api should be intuitive, abstract opengl specifics
    - should be modular -> create custom render pipilines, different shader programs appart from default should be supported
    - io, window management is not supposed to be part of the library(library works with any opengl context)
    - optimazation algorithms are not a part of library because they are case dependent, but consider adding defaults (LOD, frustum culling)
